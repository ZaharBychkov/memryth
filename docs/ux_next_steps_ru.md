# MEMRYTH: UX-заметки для следующего прохода

Дата фиксации: 2026-05-04

Обновлено: 2026-05-05

Цель файла: быстро восстановить контекст следующего рабочего чата и продолжить
без повторного обсуждения уже принятых решений.

Статус: ближайший UX-проход выполнен. Этот файл оставлен как журнал решений и
как список остаточных UX/архитектурных долгов.

## 1. Что показал первый пользовательский прогон

Пользователь впервые открыл приложение и сразу нажал на фильтр `Цитата`.
Он не понял, что это фильтр, а не действие создания записи.

Вывод:

- строка `Цитата / Мысль / Фрагмент / ★` сейчас может читаться как
  навигация или создание записи;
- особенно плохо это работает на пустой библиотеке, когда фильтровать еще
  нечего;
- пустой старт должен вести к первому действию, а не показывать инструменты
  управления библиотекой.

## 2. Пустая библиотека

Статус: выполнено.

Самый сильный вариант: если `totalCount == 0`, показывать не обычный экран
библиотеки, а экран первого действия.

Скрыть на пустом старте:

- строку `[Сортировка] [Массовые] [Темы] [Тема] [Настройки]` частично или
  полностью;
- поиск;
- фильтры `Цитата / Мысль / Фрагмент / ★`;
- массовые действия;
- индекс тем.

Оставить:

- заголовок `MEMRYTH`;
- доступ к настройкам, потому что язык, тема и импорт могут понадобиться до
  первой записи;
- центральный текст вроде `Ваша библиотека пока пуста`;
- короткое пояснение: `Сохраните первую мысль, цитату или фрагмент`;
- основную кнопку `Создать первую запись`;
- вторичную кнопку `Импортировать backup`, потому что у пользователя уже может
  быть JSON.

После появления первой записи включать обычный интерфейс библиотеки:

- сортировка;
- поиск;
- фильтры;
- темы;
- массовые действия.

Это снижает когнитивную нагрузку и убирает сценарий, где пользователь нажимает
на неработающие или непонятные инструменты пустой библиотеки.

## 3. Фильтры типов

Статус: выполнено.

После появления записей фильтры лучше сделать более очевидными.

Идеи:

- использовать множественное число: `Цитаты`, `Мысли`, `Фрагменты`;
- добавить `Все`, чтобы строка читалась как фильтрация:
  `Все / Цитаты / Мысли / Фрагменты / ★`;
- не добавлять длинный объясняющий текст рядом с фильтрами, если смысл можно
  передать самим UI.

## 4. Форма новой записи

Статус: выполнено.

Проблема из пользовательского прогона:

- пользователь увидел первое большое поле `Текст записи`;
- начал вводить туда и автора, и источник, и сам текст;
- значит название поля слишком общее.

Решение:

- для типа `Цитата`: поле должно называться `Текст цитаты`;
- для типа `Мысль`: `Текст мысли`;
- для типа `Фрагмент`: `Текст фрагмента`.

Порядок полей пока лучше не менять радикально:

1. тип записи;
2. главный текст;
3. блок контекста;
4. заметка;
5. темы;
6. сохранение.

Почему не ставить автора и источник первыми:

- пользователь пришел сохранить содержание;
- автор и источник часто необязательны;
- если начать форму с контекста, пользователь может застрять на вторичных
  полях.

Но блок `Контекст` нужно сделать визуально понятнее:

- `Автор / собеседник`;
- `Источник`;
- эти поля должны читаться как необязательное уточнение, а не как часть
  основного текста.

## 5. Лишнее отдельное поле источника

Статус: выполнено.

Решение по направлению:

- в UI это поле не нужно;
- пользователь не понимает, зачем рядом с `Источник` есть второе похожее
  поле;
- книгу, видео, статью, страницу, главу, таймкод или ссылку можно указать в
  одном поле `Источник`;
- поле `Источник` можно переименовать/расширить подсказкой:
  `Книга, видео, статья, страница или ссылка`.

Практичный вариант реализации принят:

- удалить лишнее отдельное поле источника из пользовательской формы;
- удалить его из карточек/детального экрана;
- удалить поле из модели `Quote`, JSON export/import и Hive adapter;
- не переиспользовать Hive field id `9`, чтобы старые локальные записи не
  начали читать старые данные как другое поле.

## 6. Что уже удалено

Функция `Сохранить текущий вид` / saved filters удалена полностью, потому что:

- пользователь не понимал, за что отвечает кнопка;
- она появлялась после выбора фильтра и путала сценарий;
- диалог ввода имени вызывал Flutter assertion crash;
- для beta это слишком сложная и лишняя функция.

Удалено:

- UI saved filters;
- модель `SavedFilter`;
- строки интерфейса;
- методы сохранения/применения фильтра;
- тесты saved filters.

В настройках оставлен только obsolete-key `savedFilters`, чтобы приложение
удалило старое значение из Hive.

## 7. Технические долги для будущих правок

Зафиксированные code review findings:

1. Архитектура сейчас скорее MVVM-lite, не Clean Architecture.
   `QuoteListViewModel` просто наследует `QuoteController`. Нужно решить,
   нужен ли отдельный ViewModel-слой или лучше переименовать/упростить.

2. `AppSettings` импортирует controller ради `QuoteSortMode`.
   Лучше вынести `QuoteSortMode` в `models` или `domain`, чтобы settings не
   зависел от controller-слоя.

3. Папка `lib/contollers` содержит опечатку.
   Код работает, но лучше переименовать в `lib/controllers` отдельным
   механическим коммитом с обновлением импортов и документации.

4. Пункт `Аккаунт / Синхронизация` убран из настроек.
   Возвращать его только вместе с реальной sync/cloud-реализацией.

5. В рабочем дереве есть untracked служебные файлы:
   `.agents/`, `.claude/`, `skills-lock.json`, `docs/monetization_plan_ru.md`.
   Нужно решить:
   - добавить `.agents/`, `.claude/`, `skills-lock.json` в `.gitignore`;
   - или осознанно закоммитить нужные project tooling файлы;
   - `docs/monetization_plan_ru.md` выглядит полезным документом и может быть
     закоммичен отдельно.
