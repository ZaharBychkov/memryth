## 1. К чему стремимся

MEMRYTH должен стать не просто списком цитат, а приватной офлайн-библиотекой личных текстовых фрагментов:

- пользователь быстро сохраняет цитаты, мысли и фрагменты;
- у каждой записи есть контекст: автор, источник, заметка, темы;
- сохраненное легко найти через поиск, фильтры, избранное и сортировки;
- данные хранятся локально, без обязательного аккаунта;
- пользователь понимает, как сохранить резервную копию и восстановить библиотеку.

Главный критерий продаваемого MVP: пользователь доверяет приложению свои данные и не боится их потерять.

## 2. Что уже есть

- Создание, редактирование и удаление записей.
- Типы записей: цитата, мысль, фрагмент.
- Автор, источник, заметка.
- Темы и избранное.
- Поиск по тексту, автору, источнику, заметке и темам.
- Поиск `#тема` как явный поиск только по темам.
- Фильтры по типам, темам и избранному.
- Индекс тем из верхней панели: все использованные темы, счетчики записей, сортировка по популярности или алфавиту, дерево для вложенных тем через `/`, переход в поиск по нажатию.
- Сортировки: новые, недавно измененные, старые, случайный порядок.
- Русский и английский интерфейс.
- Светлая и темная тема.
- Настройки чтения: тема, язык, плавный размер текста записи, плавный межстрочный интервал, показ заметки, показ автора и источника, сброс настроек.
- Локальное хранение через Hive.

## 5. Как реализовать экспорт

### Где разместить

Экспорт должен находиться в настройках, но не рядом с размером текста как еще одна мелкая визуальная настройка.

Лучше разделить настройки на секции:

1. `Данные`
   - Экспорт библиотеки
   - Импорт из файла
   - Восстановить backup
   - Дата последнего экспорта
2. `Чтение`
   - Размер текста записи
   - Межстрочный интервал
   - Тема
   - Язык
   - Показывать заметку на карточке
   - Показывать автора и источник
   - Сброс настроек
3. `Безопасность`
   - PIN/biometric lock, когда появится

Так пользователь быстро понимает, что backup/export относится к сохранности данных, а не к внешнему виду.

### Формат файла

Минимальный формат:

```json
{
  "app": "MEMRYTH",
  "schemaVersion": 1,
  "exportedAt": "2026-05-01T12:00:00.000Z",
  "tags": [],
  "quotes": []
}
```

В `tags` сохранять темы. Название поля пока лучше оставить `tags`, чтобы не ломать существующую модель Hive и будущую совместимость backup-файлов:

- `id`
- `name`

В `quotes` сохранять:

- `id`
- `text`
- `author`
- `tagIds`
- `typeKey`
- `createdAt`
- `updatedAt`
- `isFavorite`
- `sourceTitle`
- `note`

Важно сохранить `schemaVersion`, чтобы будущие версии приложения могли импортировать старые backup-файлы.

### Логика экспорта

Сервис:

- `ExportImportService`
- читает все темы и записи через репозитории;
- собирает JSON;
- сохраняет файл `memryth-backup-YYYY-MM-DD-HH-mm.json`;
- предлагает пользователю сохранить/поделиться файлом через системный Android sheet.

Для реализации на Flutter лучше использовать:

- `path_provider` для временного файла;
- `share_plus` для отправки/сохранения экспортированного файла;
- `file_picker` для выбора файла при импорте.

На Android лучше не строить экспорт вокруг прямого доступа к общей файловой системе и storage permissions. Надежнее использовать системный выбор/шаринг файла.

### Логика импорта

Первый вариант:

- пользователь выбирает `.json`;
- приложение проверяет `app`, `schemaVersion`, наличие `tags` и `quotes`;
- показывает экран подтверждения;
- предлагает режим:
  - `Объединить с текущей библиотекой`;
  - `Заменить текущую библиотеку`.

Для первого релиза можно начать с одного безопасного режима:

- `Объединить`.

Замену всей библиотеки лучше добавить после тестов, потому что это риск потери данных.

### Проверки перед импортом

Нужно проверять:

- файл действительно JSON;
- `schemaVersion` поддерживается;
- записи имеют `id` и `text`;
- даты корректно парсятся;
- темы без имени не импортируются;
- дубликаты тем по имени не создаются повторно;
- конфликтующие `id` не затирают существующие записи без явного решения.

## 6. Что делать с размером текста

Идея заменить "маленький / средний / большой" на сенсорную шкалу правильная, потому что сейчас это выглядит грубо и дешево.

Но не стоит делать slider от 1 до 40. Пользователь может выставить некрасивое значение и сломать читаемость.

Лучший вариант:

- плавный slider в узком читабельном диапазоне;
- подписи не как основной UI, а как короткое текущее значение;
- живой preview карточки;
- диапазон примерно: `18`-`28`;
- дефолт: `22`.

Для межстрочного интервала:

- тоже плавный slider в узком диапазоне;
- диапазон примерно: `1.25`-`1.65`;
- дефолт: `1.4`.

Размер интерфейса, плотность карточек и количество строк свернутой карточки не добавлять в настройки на этом этапе. Это делает экран тяжелее, а ценность для минимального продукта слабая.

## 7. Сброс настроек

Сброс нужен. Он должен быть внизу настроек, отдельно от обычных параметров.

Минимально:

- "Сбросить настройки интерфейса";
- подтверждение через dialog;
- сбрасываются тема, язык, размер текста, межстрочный интервал и preview-параметры карточек;
- пользовательские записи и темы не трогаются.

Текст должен ясно говорить:

`Записи, темы и заметки не будут удалены. Сбросятся только настройки внешнего вида и чтения.`

Удаление всей библиотеки не добавлять в первый релиз. Если когда-нибудь понадобится, оно должно быть отдельно, с очень жестким подтверждением.

## 8. Что делать бесплатным, а что Pro

Бесплатно:

- базовый export всей библиотеки;
- базовый import/restore;
- импорт только в безопасном режиме merge без перезаписи текущей библиотеки;
- сброс настроек;
- настройки чтения;
- создание и поиск записей;
- PIN/biometric lock;
- пустой приватный старт без автоматических demo-записей.

Pro:

- export selected;
- batch export;
- автоматические backup reminders;
- widgets;
- batch actions;
- saved collections;
- resurfacing.

Причина: базовый export, import и PIN нельзя запирать за оплату.
Для приватного локального продукта это вопрос доверия, а не premium-роскошь.

Не добавлять в первый релиз:

- удаление всей библиотеки;
- режим импорта "заменить текущую библиотеку";
- автоматическое заполнение чужими demo-записями.

## 9. Будущий subscription-слой

Текущая коммерческая модель: `free + one-time Pro unlock`.

Подписку не добавлять, пока приложение остается полностью локальным и не
создает постоянных расходов на пользователя.

Отдельный subscription tier можно рассматривать позже, только если появятся:

- sync между устройствами;
- cloud backup;
- AI-функции;
- web clipper / browser extension;
- серверная обработка, хранение или регулярный внешний сервис.

Этот слой должен быть отдельным от локального Pro. Пользователь, купивший
one-time Pro, не должен терять локальные Pro-функции из-за появления подписки.
